# 固定三环境部署与修复策略

本文档记录当前版本已经落地的三环境策略。它不是未来规划稿，而是对
`environment_contract.py`、`deploy_manager.py`、`system_manager.py`、
`init_manager.py` 当前行为的整理。

## 1. 当前目标

当前策略的核心目标有三点：

1. 固定路径，避免搜索和猜测
2. 固定角色，避免 `base`、训练环境、转换环境互相污染
3. 固定合同，保证部署工具和 Web 端检查/修复使用同一套规则

## 2. 固定路径

当前固定路径不是旧文档中的 `/root/miniforge3` 和 `/root/MetaQA_CloudTraining`，
而是统一以登录用户家目录下的 `cloud-training-runtime` 为根。

| 项目 | 默认路径 |
|------|----------|
| 运行时根目录 | `~/cloud-training-runtime` |
| conda | `~/cloud-training-runtime/miniforge3/bin/conda` |
| Web Python | `~/cloud-training-runtime/miniforge3/bin/python` |
| 训练 Python | `~/cloud-training-runtime/miniforge3/envs/cloud-training/bin/python` |
| 转换 Python | `~/cloud-training-runtime/miniforge3/envs/cloud-conversion/bin/python` |
| 项目目录 | `~/cloud-training-runtime/MetaQA_CloudTraining` |
| 部署状态文件 | `~/cloud-training-runtime/MetaQA_CloudTraining/data/deploy_state.json` |

在服务端运行时，项目内部数据路径仍由 `config/app_config.yaml` 解析，默认写入
仓库目录下的 `./data`。

## 3. 环境角色

当前三环境角色如下。

| 角色 | 当前实现 | 约束 |
|------|----------|------|
| `base(web)` | 运行 `run.py`、FastAPI、系统状态接口 | 不承载训练和转换主流程 |
| `cloud-training` | 执行训练、导出 ONNX、训练期监控 | 禁止出现 `tensorflow/onnx2tf` |
| `cloud-conversion` | 执行 TFLite 转换和转换门禁验证 | 不承担 Web 服务 |

当前调用方式必须遵守以下规则：

- Web 启动使用固定 `base` Python
- 训练子进程使用固定训练环境 Python
- 转换子进程使用固定转换环境 Python
- 固定环境缺失时直接报错，不回退到系统 Python

## 4. 共享合同层

当前三环境策略的真正单一来源是
`app/core/environment_contract.py`。

它定义了以下内容：

| 合同项 | 当前内容 |
|--------|----------|
| `RUNTIME_ROOT` | `~/cloud-training-runtime` |
| `RUNTIME_ENV_REQUIREMENTS` | 三份 requirements 锁定文件路径 |
| `TRAINING_IMPORT_CHECK_SNIPPET` | 训练环境导入验证脚本 |
| `CONVERSION_IMPORT_CHECK_SNIPPET` | 转换环境导入验证脚本 |
| `CONVERSION_GATE_SNIPPET` | 与转换导入验证保持一致 |
| `ENV_ISOLATION_CHECK_SNIPPET` | 检查训练环境中是否存在 `tensorflow/onnx2tf` |
| `RUNTIME_REPAIR_STEPS` | Web 端修复任务的 9 个固定步骤 |

部署工具、系统检查、自动修复都从这份合同层取值，所以后续文档更新也必须以
它为准。

## 5. 部署链路

当前部署链路已经稳定为 12 步，而不是旧文档中的 9 步。

1. 连接服务器
2. 环境预检
3. 初始化目录与环境
4. 停止旧服务
5. 上传文件
6. 安装 Web 依赖
7. 创建训练环境
8. 创建转换环境
9. 验证固定环境
10. `base` 瘦身与清理
11. 配置自启动
12. 启动并验证服务

### 5.1 上传策略

当前部署使用 zip 全量上传覆盖：

- 本地收集文件并打包 zip
- 上传到远端临时目录
- 保留远端 `data` 目录
- 清理旧代码并解压覆盖

当前实现已经不再使用 MD5 文件级断点续传。

### 5.2 自启动策略

当前部署使用 `systemd`，服务名为 `metaqa-cloud.service`。
旧文档中的 Supervisor 方案已废弃。

## 6. deploy_state.json

部署工具当前通过 `deploy_state.json` 判断是否需要重新做 `base` 瘦身和环境快照。

当前实际关键字段如下：

```json
{
  "schema_version": 1,
  "env_layout": "base-web__cloud-training__cloud-conversion",
  "project_bootstrapped": true,
  "base_slimmed": true,
  "base_slimmed_at": "2026-05-17T10:30:00Z",
  "last_deploy_at": "2026-05-17T10:30:00Z",
  "env_snapshot": "env-snapshot-v0.1.0-20260517-103000.json",
  "version_locks": true,
  "requirements_hashes": {
    "web": "xxx",
    "training": "xxx",
    "conversion": "xxx"
  }
}
```

### 6.1 当前重新瘦身判定

当前代码在以下任一场景下会重新进入 `base` 瘦身流程：

- 状态文件不存在
- `base_slimmed` 为假
- `base_slimmed_at` 缺失
- `env_snapshot` 缺失
- `version_locks` 缺失

也就是说，当前实现是“状态标记驱动”的保守判定，而不是旧文档里更复杂的
指纹比对策略。

## 7. base 处理边界

当前版本保留 `base` 瘦身，但边界非常明确。

### 7.1 允许的动作

- 备份 `base` 的 `pip freeze`
- 卸载训练和转换重包
- 验证 Web 依赖仍可导入
- 清理 `pip` 和 `conda` 缓存

### 7.2 不允许的动作

- 在 Web 端系统修复里修改 `base`
- 在训练/转换环境未验证前就提前瘦身
- 每次普通后续部署都重复卸载 `base` 重包

### 7.3 当前卸载目标

当前 `base` 瘦身主要清理以下重包：

- `torch`
- `torchvision`
- `torchaudio`
- `ultralytics`
- `onnx`
- `tensorflow`
- `onnx2tf`
- `onnxruntime`
- `onnxsim`
- `tf-keras`
- `h5py`
- `flatbuffers`
- `protobuf`

## 8. Web 端检查策略

当前 Web 端检查来自 `system_manager.py`，检查项顺序固定为：

1. Python 环境
2. CUDA 驱动
3. GPU 可用性
4. 训练环境
5. 转换环境
6. PyTorch
7. Ultralytics
8. ONNX
9. TensorFlow
10. onnx2tf
11. 转换门禁
12. 环境隔离
13. 磁盘空间
14. 数据目录

当前汇总状态只有三类：

- `ready`
- `partial`
- `failed`

## 9. Web 端修复策略

当前 Web 端修复来自 `init_manager.py`，修复步骤和共享合同层完全一致。

1. 检查固定路径
2. 检查训练环境
3. 检查转换环境
4. 创建缺失环境
5. 同步训练依赖
6. 同步转换依赖
7. 验证训练环境
8. 验证转换环境
9. 重新汇总检查结果

### 9.1 当前修复边界

当前系统页修复只允许：

- 创建缺失训练环境
- 创建缺失转换环境
- 同步训练锁定依赖
- 同步转换锁定依赖
- 重新验证和汇总结果

当前系统页修复不会：

- 停止 Web 服务
- 修改 `base` 重依赖
- 重新定义环境角色
- 扫描候选 Python 路径

## 10. 修复任务与日志

当前修复任务具有明确的任务态和日志文件。

| 项目 | 当前实现 |
|------|----------|
| 任务状态 | `queued`、`repairing`、`success`、`failed` |
| 当前步骤 | `current_step` |
| 百分比 | `percent` |
| 耗时 | `elapsed_seconds` |
| 步骤列表 | `steps[]` |
| 最近日志 | `logs[]` |
| 日志下载 | `/api/v1/system/fix/{task_id}/log` |

日志落盘规则如下：

- 当前任务日志：`data/logs/init-fix-<task_id>.log`
- 最新镜像日志：`data/logs/init-fix.log`

前端系统页已经提供“下载日志”按钮。

## 11. 可观测性

当前版本已经把“慢步骤可见性”做到了两条链路中。

### 11.1 部署工具

部署工具会输出：

- 当前步骤
- 当前子阶段说明
- requirements 心跳日志
- 启动失败时的自动诊断信息

### 11.2 Web 端修复

系统页会展示：

- 当前修复步骤
- 进度条
- 两列步骤状态
- 最近日志
- 已耗时秒数

前端轮询频率当前约为 1.5 秒。

## 12. 当前异常处理边界

当前实现的失败边界如下。

| 场景 | 当前处理 |
|------|----------|
| 训练环境创建失败 | 部署终止，不进入 `base` 瘦身 |
| 转换环境创建失败 | 部署终止，不进入 `base` 瘦身 |
| `base` 瘦身校验失败 | 部署终止，不写完整成功标记 |
| 服务启动失败 | 采集诊断日志并停止后续成功声明 |
| Web 修复失败 | 保持 Web 在线，任务状态置为 `failed` |

## 13. 与旧策略的差异

当前版本已经与旧文档中的若干设计产生明确差异：

- 固定路径改为 `~/cloud-training-runtime`
- 不再使用 `/root/MetaQA_CloudTraining`
- 不再把 Supervisor 作为主自启动方案
- 不再把 MD5 文件级断点续传作为上传主策略
- Web 端修复步骤改为共享合同驱动的 9 步
- 重新瘦身判定当前主要依赖 `deploy_state.json` 标记

## 14. 当前验收口径

以当前代码为准，三环境策略已经满足以下验收点：

1. Web、训练、转换三环境有固定路径和固定角色
2. 部署工具与 Web 端共用同一套环境合同
3. Web 修复不会修改 `base`
4. 部署成功后会生成 `deploy_state.json` 和带版本号的环境快照
5. 系统页可以展示修复步骤、日志和下载入口
