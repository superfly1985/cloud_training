# 系统初始化模块

本文档说明当前版本的系统环境检查与自动修复实现。当前版本不再使用
独立的“强制初始化页”或 `/init` 路由守卫，而是把环境能力统一收敛到
**系统状态页**、顶部状态指示器，以及部署工具共用的环境合同层。

## 1. 当前设计目标

当前实现的目标是把运行时环境收敛为固定路径、固定角色、固定检查项，
并让 Web 端检查和部署工具预检使用同一套约束。

- 固定运行时根目录为 `~/cloud-training-runtime`
- 固定三环境为 `base(web)`、`cloud-training`、`cloud-conversion`
- Web 端只负责检查和修复训练/转换环境，不修改 `base`
- 检查结果统一汇总为 `ready`、`partial`、`failed`
- 修复过程提供步骤、进度、耗时、实时日志和日志下载

## 2. 固定路径与环境角色

当前版本的固定路径由 `app/core/environment_contract.py` 和
`app/core/system_manager.py` 统一定义，默认指向登录用户家目录下的运行时根。

| 角色 | 默认路径 | 用途 |
|------|----------|------|
| Web Python | `~/cloud-training-runtime/miniforge3/bin/python` | 运行 `run.py`、Web 接口、系统检查 |
| 训练环境 | `~/cloud-training-runtime/miniforge3/envs/cloud-training/bin/python` | 训练、ONNX 导出、训练监控 |
| 转换环境 | `~/cloud-training-runtime/miniforge3/envs/cloud-conversion/bin/python` | TFLite 转换、转换门禁验证 |
| conda | `~/cloud-training-runtime/miniforge3/bin/conda` | 创建和维护固定环境 |
| 运行时目录 | `~/cloud-training-runtime/MetaQA_CloudTraining` | 部署后的项目根目录 |

如果 `config/app_config.yaml` 中显式配置了 `server.python_cmd`、
`training.python_cmd` 或 `convert.python_export_cmd`，运行时会优先使用配置值。

## 3. 当前检查项

当前检查项与 `system_manager.py` 中的 `CHECK_ITEM_NAMES` 完全一致，顺序如下。

| 检查项 | 说明 | 自动修复 | 阻断 |
|--------|------|----------|------|
| Python 环境 | 固定 Web Python 是否存在且可运行 | 否 | 是 |
| CUDA 驱动 | `nvidia-smi` 是否可用 | 否 | 是 |
| GPU 可用性 | GPU 是否可访问 | 否 | 是 |
| 训练环境 | 固定训练环境 Python 是否存在 | 是 | 是 |
| 转换环境 | 固定转换环境 Python 是否存在 | 是 | 是 |
| PyTorch | 训练环境中 `torch` 可导入且 CUDA 可用 | 是 | 是 |
| Ultralytics | 训练环境中 `ultralytics` 可导入 | 是 | 是 |
| ONNX | 训练环境中 `onnx` 可导入 | 是 | 是 |
| TensorFlow | 转换环境中 `tensorflow` 可导入 | 是 | 是 |
| onnx2tf | 转换环境中 `onnx2tf` 可导入 | 是 | 是 |
| 转换门禁 | `YOLO + TensorFlow + onnx2tf` 导入链可通过 | 否 | 是 |
| 环境隔离 | 训练环境中不能出现 `tensorflow/onnx2tf` | 是 | 是 |
| 磁盘空间 | 根分区剩余空间是否大于 8 GB | 否 | 否，警告项 |
| 数据目录 | `datasets_path` 是否存在 | 是 | 否，警告项 |

## 4. 汇总状态

当前汇总逻辑由 `get_environment_summary()` 统一处理，不再使用旧文档里的
`unknown/checking/repairing` 页面态来驱动路由。

| 汇总状态 | 含义 | 触发条件 |
|----------|------|----------|
| `ready` | 系统正常 | 所有检查项通过 |
| `partial` | 部分检查需处理 | 仅存在 `warning`，或仅存在非阻断异常 |
| `failed` | 存在阻断项异常 | 任意阻断检查项为 `fail` |

系统检查任务本身的执行态仍会返回 `queued`、`checking`、`success`、`failed`，
供前端轮询展示。

## 5. 自动修复流程

当前自动修复流程完全复用共享合同层中的 `RUNTIME_REPAIR_STEPS`，步骤如下。

1. 检查固定路径
2. 检查训练环境
3. 检查转换环境
4. 创建缺失环境
5. 同步训练依赖
6. 同步转换依赖
7. 验证训练环境
8. 验证转换环境
9. 重新汇总检查结果

当前实现有三个关键边界：

- 只会操作固定训练环境和固定转换环境
- 不会在 Web 端修复流程里修改 `base` 重依赖
- 修复完成后会重新运行环境检查并写回最新状态

## 6. 依赖合同

当前 Web 端检查和修复与部署工具使用同一份合同定义。

| 合同项 | 当前值 |
|--------|--------|
| Web 锁定文件 | `deploy_tool/requirements-web.txt` |
| 训练锁定文件 | `deploy_tool/requirements-training.txt` |
| 转换锁定文件 | `deploy_tool/requirements-conversion.txt` |
| 训练验证脚本 | `import torch, ultralytics, onnx` |
| 转换验证脚本 | `from ultralytics import YOLO; import tensorflow; import onnx2tf` |
| 隔离检查 | 训练环境中 `tensorflow/onnx2tf` 必须缺失 |

这意味着文档、部署工具、系统状态页三者看到的环境合同必须保持同一套口径。

## 7. 当前 API

当前版本主要通过 `/api/v1/system/*` 提供环境检查和修复能力，前端系统页
直接调用这组接口。

### 7.1 系统检查与修复接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/system/status` | GET | 获取顶部状态栏和系统概览数据 |
| `/api/v1/system/checks` | GET | 获取最近一次环境检查任务和检查项 |
| `/api/v1/system/checks` | POST | 启动新的环境检查任务 |
| `/api/v1/system/fix` | POST | 启动统一自动修复 |
| `/api/v1/system/fix/current` | GET | 获取当前修复任务 |
| `/api/v1/system/fix/{task_id}` | GET | 查询指定修复任务状态 |
| `/api/v1/system/fix/{task_id}/log` | GET | 下载修复日志 |

### 7.2 兼容保留接口

`app/api/init.py` 仍保留一组较轻量的初始化接口，但当前前端主流程已经不再依赖
它们进行页面守卫。

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/init/run` | POST | 触发一次初始化检查 |
| `/api/v1/init/auto-fix` | POST | 启动自动修复 |
| `/api/v1/init/auto-fix/{task_id}` | GET | 查询修复任务 |

### 7.3 检查项结构

系统检查项的返回结构与当前实现一致。

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 检查项名称 |
| `status` | string | `pass`、`fail`、`warning` |
| `message` | string | 检查结果说明 |
| `auto_fixable` | bool | 是否可纳入统一修复 |
| `blocking` | bool | 是否属于阻断项 |

### 7.4 修复任务结构

修复任务返回的是任务视图，而不是旧文档中的单条进度状态。

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | string | 修复任务 ID |
| `status` | string | `queued`、`repairing`、`success`、`failed` |
| `current_step` | string | 当前步骤名称 |
| `current_step_index` | int | 当前步骤序号，从 1 开始 |
| `total_steps` | int | 总步骤数，当前为 9 |
| `percent` | int | 进度百分比 |
| `elapsed_seconds` | int | 已耗时秒数 |
| `steps` | array | 每一步的状态列表 |
| `logs` | array | 最近日志行 |
| `log_path` | string | 当前日志文件路径 |
| `summary` | object | 修复结束后的环境汇总 |

## 8. 前端表现

当前版本的“系统状态”页位于主应用标签页内部，而不是单独的全屏初始化页面。

当前界面包含以下区域：

- GPU 状态卡片
- 磁盘与基础版本信息卡片
- 环境检查列表
- 自动修复按钮与重新检查按钮
- 修复任务进度条
- 两列修复步骤清单
- 实时日志区域
- 日志下载入口

当前交互逻辑如下：

1. 进入系统页后加载系统状态和最近一次检查结果
2. 点击“重新检查”后启动 `/api/v1/system/checks`
3. 前端以约 1.5 秒轮询检查任务结果
4. 点击“自动修复”后启动 `/api/v1/system/fix`
5. 修复期间展示当前步骤、百分比、耗时和最近日志
6. 修复完成后自动刷新检查结果与顶部状态

## 9. 顶部状态指示器

顶部状态栏会综合 `system/status` 与修复任务状态，让用户不用进入系统页也能
看到环境是否异常。

| 显示状态 | 触发条件 |
|----------|----------|
| 系统正常 | 最近汇总状态为 `ready` |
| 部分检查需处理 | 汇总状态为 `partial` |
| 阻断项异常 | 汇总状态为 `failed` |
| 环境修复中 | 当前存在 `queued/repairing` 修复任务 |

修复中优先级最高，会覆盖普通健康状态文本。

## 10. 状态持久化与日志

当前状态文件和日志路径按 `config/app_config.yaml` 解析，默认位于项目根目录
下的 `data` 目录，而不是旧文档中的 `data/config`。

| 文件 | 默认路径 | 说明 |
|------|----------|------|
| 初始化状态 | `./data/init_state.json` | 记录最近一次修复后的汇总状态 |
| 最新修复日志 | `./data/logs/init-fix.log` | 最近一次修复日志镜像 |
| 归档修复日志 | `./data/logs/init-fix-fix-xxx.log` | 按任务 ID 归档的修复日志 |

`init_state.json` 当前写入的核心字段如下：

```json
{
  "updated_at": "2026-05-17T10:30:00+00:00",
  "status": "ready",
  "statusText": "系统正常",
  "last_task_id": "fix-001"
}
```

## 11. 与旧方案的主要差异

为了避免后续再按旧方案理解，这里明确列出当前版本已经发生的变化。

- 不再使用独立的 `/init` 页面和强制路由守卫
- 不再描述“清理 base 环境冗余包”属于 Web 端修复步骤
- 不再存在“缓存清理”这一独立检查项
- 数据目录检查改为读取配置解析后的 `datasets_path`
- 修复流程与部署工具共享合同层，不再各自维护一套依赖规则
