# 部署方案

本文档描述当前版本的真实部署实现。以部署工具
`deploy_tool/deploy_manager.py` 和 `deploy_tool/deploy_gui.py` 为准，
不再沿用旧文档中的 `/opt/cloud-training`、`MD5 断点续传`、
`Supervisor` 或 `9 步部署` 设计。

## 1. 部署目标

当前部署方案的目标是把远端服务器固定为同一套运行时布局，并通过 Windows
桌面部署工具完成一键部署、环境补齐、自启动配置和服务验证。

部署工具当前具备以下特征：

- 通过 SSH 连接远程 Linux 服务器
- 自动解析登录用户的 `HOME` 目录
- 在 `~/cloud-training-runtime` 下安装 Miniforge 和项目运行时
- 维护 `base(web)`、`cloud-training`、`cloud-conversion` 三环境
- 使用 `systemd` 配置自启动
- 用 zip 全量上传覆盖替代文件级增量对比

## 2. 目标目录布局

当前远端目录并不是固定写死到 `/root` 或 `/opt`，而是基于登录用户的家目录
动态展开。

| 项目 | 默认路径 |
|------|----------|
| 运行时根目录 | `~/cloud-training-runtime` |
| 项目目录 | `~/cloud-training-runtime/MetaQA_CloudTraining` |
| conda | `~/cloud-training-runtime/miniforge3/bin/conda` |
| Web Python | `~/cloud-training-runtime/miniforge3/bin/python` |
| 训练 Python | `~/cloud-training-runtime/miniforge3/envs/cloud-training/bin/python` |
| 转换 Python | `~/cloud-training-runtime/miniforge3/envs/cloud-conversion/bin/python` |
| 部署状态文件 | `~/cloud-training-runtime/MetaQA_CloudTraining/data/deploy_state.json` |

如果用户是 `root`，上述 `~` 会解析为 `/root`。如果用户是 `ubuntu`，则会解析为
`/home/ubuntu`。

## 3. 三环境角色

当前部署工具和 Web 端环境检查都遵循同一套三环境分工。

| 环境 | 用途 | 约束 |
|------|------|------|
| `base(web)` | 运行 `run.py`、Web API、日志与状态接口 | 不承载训练和转换大包 |
| `cloud-training` | 训练、ONNX 导出、训练期监控 | 不允许出现 `tensorflow/onnx2tf` |
| `cloud-conversion` | TFLite 转换、转换门禁验证 | 不承担 Web 服务 |

当前锁定文件如下：

- `deploy_tool/requirements-web.txt`
- `deploy_tool/requirements-training.txt`
- `deploy_tool/requirements-conversion.txt`

## 4. 环境要求

当前方案面向 Linux 服务器，部署工具会尽量自动安装依赖，但以下条件仍然是前提。

| 项目 | 要求 | 说明 |
|------|------|------|
| 操作系统 | Linux，优先 Ubuntu | 当前命令以 `apt-get` 为主 |
| SSH | 可通过用户名和密码登录 | 部署工具使用 Paramiko |
| sudo | 推荐具备免交互 `sudo -n` | 用于安装系统库和写入 `systemd` |
| GPU | 推荐 NVIDIA GPU | 训练和 CUDA 检查依赖 `nvidia-smi` |
| 磁盘空间 | 建议保留 8 GB 以上可用空间 | 低于阈值时系统页会告警 |

## 5. 当前 12 步部署流程

当前部署工具的 `DEPLOY_STEPS` 如下，文档和界面都应保持这 12 步口径。

1. 连接服务器
2. 环境预检
3. 初始化目录与环境
4. 停止旧服务
5. 上传文件
6. 安装 Web 依赖
7. 创建训练环境
8. 创建转换环境
9. 验证固定环境
10. `base` 瘦身与清理
11. 配置自启动
12. 启动并验证服务

### 5.1 环境预检

当前预检会检查以下信息，并据此生成执行计划：

- 固定 Web Python 是否存在
- conda 是否存在
- 训练环境和转换环境是否存在
- 系统运行库 `libgl1`、`libglib2.0-0` 是否存在
- `systemd` 是否可用
- 远端 `deploy_state.json` 中记录的 requirements 哈希
- 本地三份 requirements 锁文件的 MD5

### 5.2 运行时初始化

初始化步骤会完成以下动作：

1. 创建 `~/cloud-training-runtime`
2. 读取远端 `deploy_state.json`
3. 判断是否需要重新执行 `base` 瘦身
4. 缺少 Miniforge 时自动下载安装
5. 创建项目根目录
6. 缺少系统运行库时尝试执行 `sudo -n apt-get install`

## 6. 文件上传策略

当前版本已经不再按文件做 MD5 断点续传，而是改为 **本地打 zip，远端单包上传，
解压后覆盖代码目录**。

当前行为如下：

1. 收集允许部署的文件列表
2. 排除 `.git`、`data`、`test`、`.trae` 等目录
3. 在本地创建临时 zip 包
4. 上传到远端 `data/temp/deploy-upload-*.zip`
5. 保留远端 `data` 目录，清空项目目录下其他代码内容
6. 用固定 Web Python 解压 zip 覆盖代码
7. 清理本地和远端临时 zip

该策略的目的很明确：

- 避免大量小文件逐个上传
- 降低文件对比逻辑复杂度
- 提升重新部署时的稳定性
- 保留远端 `data` 持久化目录不被覆盖

## 7. 依赖安装与验证

当前三类依赖的处理方式如下。

| 分类 | 安装方式 | 触发条件 |
|------|----------|----------|
| Web 依赖 | `base` 环境执行 `pip install -r requirements-web.txt` | 本地与远端哈希不一致 |
| 训练依赖 | 训练环境执行 `pip install -r requirements-training.txt` | 环境缺失或哈希不一致 |
| 转换依赖 | 转换环境执行 `pip install -r requirements-conversion.txt` | 环境缺失或哈希不一致 |

当前部署工具会在长时间安装大包时输出心跳日志，例如：

```text
仍在执行：安装 cloud-training 锁定依赖，当前依赖 6/18: ultralytics==8.3.148，已耗时 07:00
```

这套心跳日志同时会驱动 GUI 中的当前步骤说明。

### 7.1 固定环境验证

当前环境验证与共享合同层一致。

| 环境 | 验证内容 |
|------|----------|
| 训练环境 | `import torch, ultralytics, onnx` |
| 转换环境 | `from ultralytics import YOLO; import tensorflow; import onnx2tf` |

只有两个固定环境都验证通过，部署才会继续后面的步骤。

## 8. base 瘦身与清理

当前版本仍保留 `base` 瘦身，但只会在满足条件时执行，不会每次部署都重复卸载。

需要重新执行 `base` 瘦身的常见场景：

- 首次部署
- `deploy_state.json` 不存在
- `base_slimmed` 标记缺失
- `base_slimmed_at` 缺失
- `env_snapshot` 缺失
- `version_locks` 缺失

当前瘦身动作包括：

1. 备份 `base` 的 `pip freeze` 和 `pip list`
2. 卸载训练和转换重包
3. 验证 Web 依赖仍可导入
4. 清理 `pip cache`
5. 执行 `conda clean -a -y`

## 9. 自启动方案

当前版本使用 `systemd`，不再使用 Supervisor。

### 9.1 服务名

| 项目 | 当前值 |
|------|--------|
| unit 名称 | `metaqa-cloud.service` |
| 端口 | `8090` |
| 工作目录 | `~/cloud-training-runtime/MetaQA_CloudTraining` |
| 启动命令 | `~/cloud-training-runtime/miniforge3/bin/python run.py` |

### 9.2 systemd 配置特征

当前生成的 unit 具备这些关键内容：

- `WorkingDirectory` 指向项目目录
- `ExecStart` 使用固定 Web Python
- `Restart=always`
- `Environment=CT_PORT=8090`
- 标准输出和标准错误都写入 `data/logs/server.log`

### 9.3 回退启动

如果 `systemd restart` 失败，当前实现会回退到 `nohup ... run.py` 启动，随后继续
做端口和 HTTP 验证。

## 10. 服务验证

启动完成后，部署工具会按顺序做两类验证。

1. 本地 socket 检测远端 `8090` 端口是否监听
2. 远端执行 `curl http://localhost:8090/` 检查 HTTP 响应

如果启动失败，当前实现会自动收集以下诊断信息并输出到日志面板：

- `server.log`
- `systemctl status`
- `journalctl`
- 相关 Python 进程列表
- 端口监听信息

## 11. deploy_state.json

当前部署工具会写入远端部署状态文件，用于判断后续部署是否需要重做部分步骤。

示例结构如下：

```json
{
  "schema_version": 1,
  "env_layout": "base-web__cloud-training__cloud-conversion",
  "project_bootstrapped": true,
  "base_slimmed": true,
  "base_slimmed_at": "2026-05-17T10:30:00Z",
  "last_deploy_at": "2026-05-17T10:30:00Z",
  "env_snapshot": "env-snapshot-v0.1.0-20260517-103000.json",
  "version_locks": true,
  "requirements_hashes": {
    "web": "xxx",
    "training": "xxx",
    "conversion": "xxx"
  }
}
```

当前环境快照会输出到 `data/logs/`，文件名包含项目版本号。

## 12. 部署工具 GUI

当前 GUI 已经不是旧文档中的单列布局，实际界面为左右两栏加底部状态区。

### 12.1 当前布局

- 左栏：服务器连接、部署信息、操作区
- 右栏：部署步骤、部署日志
- 底部整行：部署状态、进度条、耗时

### 12.2 当前交互

| 项目 | 当前实现 |
|------|----------|
| 主按钮 | `测试连接`、`一键部署`、`取消部署` 统一宽度 |
| 步骤区 | 12 个步骤分两列显示 |
| 日志区 | 右侧大面积滚动日志面板 |
| 服务器 Web | 在“部署信息”中显示可点击链接 |
| 耗时 | 底部实时显示部署耗时 |
| 进度 | 步骤状态和底部进度条同步更新 |

### 12.3 服务器 Web 链接

当前 GUI 会根据输入的主机地址自动生成服务器访问地址：

```text
http://<host>:8090
```

部署成功后，用户可以直接点击“部署信息”中的链接在浏览器中打开 Web 页面。

## 13. 手动验证命令

如果需要手工排查当前部署结果，优先使用下列命令。以下路径都以登录用户
`HOME` 为基准。

```bash
export RUNTIME_ROOT="$HOME/cloud-training-runtime"
export APP_DIR="$RUNTIME_ROOT/MetaQA_CloudTraining"

$RUNTIME_ROOT/miniforge3/bin/python --version
$RUNTIME_ROOT/miniforge3/envs/cloud-training/bin/python -c "import torch, ultralytics, onnx; print(torch.__version__)"
$RUNTIME_ROOT/miniforge3/envs/cloud-conversion/bin/python -c "from ultralytics import YOLO; import tensorflow, onnx2tf; print('ok')"
sudo -n systemctl status metaqa-cloud.service --no-pager -n 50
curl -I http://localhost:8090/
```

## 14. 故障排查

| 问题 | 常见原因 | 当前处理方式 |
|------|----------|--------------|
| Miniforge 安装失败 | 无网络或 GitHub 下载失败 | 重试部署，或手工预装到运行时根 |
| Web 依赖重复安装很慢 | requirements 哈希变化 | 查看心跳日志中的当前依赖进度 |
| 转换环境验证失败 | `tensorflow/onnx2tf` 链未满足 | 检查转换环境锁定文件和验证日志 |
| 服务启动失败 | 端口占用或 `systemd` 配置失败 | 查看部署工具自动采集的 `server.log` 和 `journalctl` |
| 重新部署后仍执行 base 瘦身 | 状态文件缺少完整标记 | 检查 `deploy_state.json` 是否包含 `base_slimmed_at`、`env_snapshot`、`version_locks` |

## 15. 与旧文档的差异

为了避免继续按旧方案理解，当前版本已经明确改变以下行为：

- 不再使用 `/opt/cloud-training`
- 不再使用 `/root/miniforge3` 作为唯一写死路径
- 不再使用 MD5 文件级断点续传上传
- 不再使用 Supervisor 自启动
- 不再使用 9 步部署流程
- 部署 GUI 已改为左右布局、两列步骤、底部整行状态栏
