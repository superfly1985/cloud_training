# TFLite dataset YAML handoff design

本设计修复 `MetaQA_CloudTraining` 中训练完成后打包 `TFLite` 时因为缺少
`dataset.yaml` 而跳过转换的问题。目标是让训练链路把本次实际使用的
`data.yaml` 一并带入任务运行目录，并让打包链路优先使用任务目录中的
YAML，在缺失时再回退到数据集目录。

## 问题概述

当前系统在训练启动时会在数据集目录生成最终用于训练的 `data.yaml`，并
把这一路径传给训练脚本。训练完成后，打包链路在训练运行目录里查找
`data.yaml`，找不到就直接记录 “TFLite 导出失败（缺少 dataset.yaml）”。

线上实际证据表明，最新训练任务目录只包含：

- `best.pt`
- `best.onnx`
- `results.csv`
- `train.log`

运行目录里没有 `data.yaml` 或 `dataset.yaml`，因此 `onnx2tf` 根本没有进
入实际转换阶段。

## 目标结果

本改造完成后，系统要满足以下结果：

- 训练启动时会把本次最终实际使用的 `data.yaml` 复制到任务运行目录。
- 打包链路创建产物包时，优先从任务运行目录或 `train` 子目录读取
  `data.yaml` / `dataset.yaml`。
- 如果任务目录里没有 YAML，打包链路会回退到数据集目录，继续使用训练
  时的同源 YAML。
- 新产物包里会继续包含 `dataset.yaml` 文件。
- 已完成但未重新训练的旧任务在重新打包时，只要数据集目录里的 YAML 仍
  存在，也可以恢复 `TFLite` 转换。

## 设计方案

本次改造采用双保险方案，既修训练侧，也修打包侧。

### 训练侧

训练管理模块继续在数据集目录生成最终 `data.yaml`，但在生成后立即复制
一份到任务运行目录，例如：

- `run_dir/data.yaml`

复制的必须是训练时最终实际使用的 YAML，而不是上传压缩包里的原始版本。
这样训练产物目录具备自包含能力，便于后续打包、补打包和问题排查。

### 打包侧

打包模块新增 YAML 解析回退策略：

1. 优先在 `train_dir` 查找 `data.yaml`
2. 再查 `train_dir` 下的 `dataset.yaml`
3. 再查 `run_dir` 下的 `data.yaml` / `dataset.yaml`
4. 最后根据任务的 `dataset_id` 回退到数据集目录里的 `data.yaml` 或
   `dataset.yaml`

只要任一路径找到有效 YAML，就继续进入 `onnx2tf` 转换。

## 数据流约束

本节定义本次修复后的数据一致性要求。

- 训练脚本使用的 YAML 与复制到任务目录中的 YAML 必须是同一份最终内容。
- 打包时如果优先命中任务目录 YAML，就以任务目录版本为准。
- 回退到数据集目录仅作为兼容旧任务的兜底行为。
- 产物包中的 `dataset.yaml` 以本次打包最终实际使用的 YAML 为准。

## 兼容性要求

本改造必须兼容新任务和旧任务。

- 新任务训练后会自动携带运行目录 YAML。
- 旧任务没有运行目录 YAML 时，重新打包仍可通过数据集目录回退继续转换。
- 如果旧任务和对应数据集目录都没有 YAML，仍保持当前失败行为并记录明
  确错误。

## 测试策略

本次改造至少需要覆盖以下行为：

- 训练启动时会把最终 `data.yaml` 复制到任务运行目录。
- 打包时优先使用任务目录里的 YAML。
- 任务目录缺失 YAML 时，打包会回退到数据集目录里的 YAML。
- 如果两处都缺失，错误信息仍然明确为缺少 `dataset.yaml`。

## 范围限制

本次改造只修复 YAML 传递和回退策略，不包含以下内容：

- 不修改 `onnx2tf` 的转换参数。
- 不修改训练数据切分逻辑。
- 不修改产物包其他元信息结构。
