# 补传合并按同名覆盖图片与标签设计

本文档定义“补传/合并数据集”链路的新行为。目标是让补传更符合实际使用场景：
同名图片和同名标签都按覆盖处理，并且图片与标签分别独立计算，不再把标签
导入绑定在图片导入结果上。

## 背景

当前实现里，补传走 `merge_dataset()`，底层复用 `import_zip(...,
skip_existing=True)`。这导致两个明显问题：

- 同名图片会被直接跳过，而不是覆盖。
- 标签导入依赖图片导入流程，图片被跳过时，对应标签也会一起被忽略。

这个行为不符合补传的真实业务语义。补传常见场景不是“避免重复上传”，而是
“重新上传修订后的图片”或“补传更新后的标注文件”。因此，补传必须以覆盖为
主，而不是跳过为主。

## 目标

本设计要达成以下目标：

- 只修改补传/合并数据集这条链路。
- 同名图片在补传时直接覆盖旧图片。
- 同名标签在补传时直接覆盖旧标签。
- 图片和标签分别独立处理，不互相依赖是否更新。
- 不判断图片内容或标签内容是否变化，发现同名即直接覆盖。
- 补传完成后，数据集统计和 `annotated` 状态与磁盘实际内容保持一致。

## 非目标

本设计不包含以下改动：

- 不修改“新建数据集导入 ZIP”的行为。
- 不新增按文件哈希判断是否变化的逻辑。
- 不新增“补传未带标签则删除旧标签”的语义。
- 不修改 `dataset.yaml` 的导入/覆盖策略。

## 当前实现审计

当前核心逻辑在 `app/core/dataset_manager.py`：

- `merge_dataset()` 只是调用 `import_zip(target_id, zip_path,
  skip_existing=True)`
- `import_zip()` 遍历 ZIP 中的图片文件
- 每处理一张图片时，调用 `_find_label()` 去找同名 `.txt`
- 找到标签才一起移动到 `labels/`

这意味着当前标签是“挂在图片流程上的”，而不是独立处理的。补传时只要图片
被跳过，同名标签就永远不会落到目标数据集中。

## 设计原则

本次改动遵循以下原则：

1. 补传语义与新建导入语义必须分开。
2. 覆盖逻辑只按文件名判断，不引入内容比对。
3. 图片和标签必须两条链路独立处理。
4. 统计结果必须来自最终落盘文件，而不是导入过程中的临时判断。

## 行为设计

本节定义补传/合并的新契约。

### 补传时的图片处理

补传时，ZIP 中的图片文件按以下规则处理：

- 如果目标数据集里不存在同名图片，则直接导入。
- 如果目标数据集里已存在同名图片，则直接覆盖旧图片。
- 不生成重命名副本。
- 不跳过同名图片。

### 补传时的标签处理

补传时，ZIP 中的标签文件按以下规则处理：

- 如果目标数据集里不存在同名标签，则直接导入。
- 如果目标数据集里已存在同名标签，则直接覆盖旧标签。
- 不依赖对应图片是否刚在本次补传中被导入。
- 不要求图片和标签必须成对出现。

### 图片与标签独立性

补传时必须支持以下场景：

1. 只补图片，不补标签。
2. 只补标签，不补图片。
3. 图片和标签都补。
4. 只覆盖图片，保留旧标签。
5. 只覆盖标签，保留旧图片。

这里“独立”的含义是：

- ZIP 里只有图片时，只处理图片，不修改已有标签。
- ZIP 里只有标签时，只处理标签，不要求重新上传图片。
- ZIP 里图片和标签都存在时，分别各自覆盖同名目标文件。

## `dataset.yaml` 策略

本设计明确保持当前 `dataset.yaml` 策略不变。

当前逻辑是在目标数据集根目录还不存在 `dataset.yaml` 时，才从 ZIP 中复制一份
进去。补传不会用 ZIP 里的 `dataset.yaml` 覆盖已有配置。

原因如下：

- 本次需求只针对图片和标签更新。
- `dataset.yaml` 通常属于数据集整体配置，不适合在补传时静默覆盖。
- 保持不变可以避免把补传范围扩大到配置合并问题。

## 模块设计

本次改动保持模块边界稳定，不把逻辑塞回 API 层。

### `app/core/dataset_manager.py`

该模块继续负责 ZIP 导入和合并逻辑。

建议的职责拆分如下：

- `import_zip()` 保持新建导入行为不变。
- `merge_dataset()` 改为拥有自己的补传语义，而不是简单复用
  `skip_existing=True`。
- 如果需要，新增仅供 `merge_dataset()` 使用的内部帮助函数，用于：
  - 遍历并覆盖图片
  - 遍历并覆盖标签
  - 统一返回导入结果

### `app/api/upload.py`

API 层保持现状：

- `action == "merge"` 时继续调用 `merge_dataset()`
- 不增加补传业务细节

### 前端

前端上传入口和交互文案可以暂时不改。当前“合并新增数据”入口已经足够承载
新的覆盖语义。

如果后续需要优化，可再单独补充提示文案，例如明确说明“同名文件会覆盖”。

## 结果字段设计

当前 `import_zip()` 返回 `imported` 和 `skipped`。补传语义改为覆盖后，
`merge_dataset()` 不再适合用“跳过重复”表达结果。

建议补传返回结果至少包含：

- `images_imported`
- `images_overwritten`
- `labels_imported`
- `labels_overwritten`

如果为了兼容当前前端最小改动，也可以继续保留一个总量字段，例如：

- `imported`

但“跳过重复”字段不应再继续保留在补传结果中。

## 数据刷新策略

补传完成后必须重新刷新数据集统计，以磁盘最终状态为准。

需要保证以下结果正确：

- `image_count`
- `annotated_count`
- `classes`
- `total_size`

同时，`images` 表中的图片记录不能因为补传覆盖而重复插入同名记录。

如果某张图已经存在，补传只覆盖磁盘文件，则数据库里应继续复用原有图片记录；
如果某个标签后来被补传覆盖，最终 `annotated` 状态应由 `_refresh_dataset_stats()`
 基于磁盘标签文件重新计算。

## 测试设计

本次改动必须先补测试，再改实现。

### 新增回归测试

至少覆盖以下场景：

1. 补传同名图片时，目标图片文件内容被覆盖。
2. 补传同名标签时，目标标签文件内容被覆盖。
3. 补传只带图片时，旧标签仍保留。
4. 补传只带标签时，旧图片仍保留，且最终数据集保持已标注。
5. 补传时不会新增同名图片的重复数据库记录。
6. 新建导入链路仍保持原有逻辑，不受补传覆盖行为影响。

### 建议测试方式

使用临时目录和内存数据库，构造一个已有数据集：

- 先写入旧图片和旧标签
- 再构造只含新图片、只含新标签、图片加标签的 ZIP
- 调用 `merge_dataset()`
- 最后直接断言磁盘文件内容和数据库统计

## 风险与兼容性

本次改动是补传语义调整，主要风险如下：

- 如果仍然把标签处理绑定在图片处理流程上，可能出现“只补标签不生效”。
- 如果覆盖图片时误插入新数据库记录，可能造成同名图片重复行。
- 如果返回结果字段变化过大，可能影响前端既有成功提示。

兼容性方面，本设计已经明确：

- 新建导入不变
- API 路由不变
- 仅修改补传合并的内部语义

## 验收标准

本次实现完成后，必须满足以下验收标准：

1. 补传同名图片会覆盖旧图片。
2. 补传同名标签会覆盖旧标签。
3. 补传只带图片时，不删除旧标签。
4. 补传只带标签时，不要求重新上传图片。
5. 补传不会生成同名重命名副本。
6. 补传不会新增同名图片的重复数据库记录。
7. 新建导入行为保持不变。
8. 相关回归测试全部通过。
