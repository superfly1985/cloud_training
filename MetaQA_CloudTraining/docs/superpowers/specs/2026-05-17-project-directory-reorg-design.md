# 项目目录重组与遗留本地版分区设计

本文档定义当前仓库的中等强度目录整理方案。目标是把新 Web
版、旧本地版、一次性排障材料和历史备份明确分区，同时保证旧本地版在迁移后
仍然可以继续运行和打包。

## 背景

当前仓库内混杂了三类内容：

- 当前主产品的 Web 版，主要位于 `MetaQA_CloudTraining/`
- 仍需保留可运行能力的旧本地版，主要位于根目录 `main.py`、`src/`、
  `cloud_training.spec` 等位置
- 大量一次性排障脚本、测试产物、历史备份与旧文档，主要位于根目录
  `test/`、`backup/` 等位置

这会带来几个直接问题：

- 难以一眼判断当前主工作区和主入口
- 新旧版本入口混在根目录，容易误用
- 排障产物和历史文件长期堆积，噪音越来越高
- README、打包、测试和文档难以统一指向当前主线

用户已经确认这次整理的边界：

- 采用中等整理，不做激进重构
- 旧本地版迁移后仍要可运行、可打包
- 无用的一次性文件和重复产物倾向直接删除，而不是全部归档保留

## 目标

本次目录重组要达成以下目标：

- 明确 Web 版是当前主产品
- 将旧本地版迁移到独立目录，并保持可运行和可打包
- 清理根目录中明显无用的测试产物、探针结果和历史备份
- 让根目录保留导航级结构，而不是继续承担所有历史包袱
- 更新 README 和必要文档，让入口与目录职责一眼可见

## 非目标

本次整理不包含以下内容：

- 不把 Web 版整体重构为 `apps/web`
- 不重写 Web 版内部模块结构
- 不重构旧本地版业务代码，只做迁移与路径修正
- 不保留所有历史文件的完整归档副本
- 不在本轮处理打包产物发布目录

## 方案对比

本次确认采用“中等整理”的双主线分区方案。

### 方案 A：双主线重组 + 明确清理（最终采用）

保留两条主线：

- `MetaQA_CloudTraining/` 继续作为 Web 主线
- `legacy/local_desktop/` 作为旧本地版主线

同时对根目录 `test/`、`backup/` 中的无用内容做删除或收缩。

优点：

- 风险可控
- 不打乱当前 Web 版开发节奏
- 旧本地版仍可直接运行与打包
- 根目录清理效果明显

缺点：

- 仓库里仍保留两套产品线

### 方案 B：只搬旧本地版，不清理历史材料

优点：

- 操作更稳

缺点：

- 根目录噪音仍然很大
- 问题只解决一半

### 方案 C：直接升级为 `apps/web` + `apps/desktop`

优点：

- 长期结构最整齐

缺点：

- 改动面过大
- 短期容易影响现有路径、文档和打包链路

## 最终设计

本次采用 **方案 A**。

### 顶层目录目标

整理后，仓库根目录只保留导航级目录和少量公共文件。

目标结构如下：

```text
03.云端训练/
├─ MetaQA_CloudTraining/        # Web 主产品
├─ legacy/
│  └─ local_desktop/           # 旧本地版，可运行可打包
├─ test/                       # 仍有价值的测试与探针
├─ assets/                     # 公共资源（按需保留）
├─ README.md
├─ changelog.md
└─ 少量必要配置文件
```

根目录不再同时直接承载旧本地版入口和旧目录树。

### Web 版定位

`MetaQA_CloudTraining/` 继续作为当前主产品目录。

本次不对它的核心结构做大搬迁，只做必要的文档与说明修正，避免影响：

- 部署工具
- Web 版测试
- 当前文档
- 当前开发路径习惯

### 旧本地版定位

旧本地版整体迁移到：

```text
legacy/local_desktop/
```

迁移内容至少包括：

- 根目录 `main.py`
- 根目录 `src/`
- `cloud_training.spec`
- `cloud_training_config.json`
- 旧本地版所需依赖文件和脚本

迁移后的要求：

- 仍可直接运行
- 仍可使用原有 PyInstaller onedir 方式打包
- 旧版入口路径在 README 中清晰可见

### 测试与排障目录策略

根目录 `test/` 不再作为“任何一次性内容的堆放区”。

保留原则如下：

- 保留仍会复用的测试脚本
- 保留仍有价值的排障脚本
- 删除明显一次性的探针结果文件
- 删除重复模型产物、图片输出和缓存目录

优先删除对象包括：

- `__pycache__/`
- 一次性 `.json` 探针结果文件
- 大体积 `.tflite` 对比产物
- 临时图片输出目录
- 已无后续复用价值的日志文件

如果某个脚本仍有复用价值，但当前位置混乱，可以迁移到更明确的子目录，例如：

- `test/probes/`
- `test/manual/`
- `test/samples/`

### 历史备份策略

当前 `backup/` 中的内容不再默认保留在主工作视野中。

处理原则如下：

- 对确有参考价值的历史文档，迁移到更明确的归档位置
- 对重复备份、旧图片、旧 drawio 导出物、过期审计材料，直接删除
- `backup/old_version/` 下如果内容和 `legacy/local_desktop/` 重复，则以
  `legacy/local_desktop/` 为准，避免双份旧代码并存

## 路径与运行约束

旧本地版迁移时，必须先检查以下问题：

- 是否存在依赖当前根目录的相对路径
- 是否存在对 `src/` 的隐式导入假设
- `.spec` 文件中的入口和资源路径是否写死
- 配置文件默认位置是否依赖旧根目录

如果存在这些情况，本次整理必须先修正路径，再执行迁移。

验收标准不是“文件搬过去了”，而是：

- 旧本地版能从新位置正常启动
- 旧本地版能从新位置正常打包

## 删除规则

本次不采用“先全部归档”的保守策略，而采用规则化删除。

删除必须满足以下任一条件：

1. 文件为缓存或编译产物
2. 文件为一次性排障结果，且已经不参与当前主流程
3. 文件与已保留版本重复，且不再作为唯一参考源
4. 文件为大体积样本或中间产物，对后续开发无实际帮助

以下内容默认不得直接删除：

- 当前 Web 版生产代码
- 旧本地版唯一入口和依赖文件
- 当前仍被文档或打包链路引用的文件
- 用户明确保留的业务资源

## README 与文档策略

根目录 `README.md` 必须在这次整理中同步更新。

至少要说明以下信息：

- 当前主产品是 Web 版，位于 `MetaQA_CloudTraining/`
- 旧本地版位于 `legacy/local_desktop/`
- 旧本地版仍可运行和打包
- `test/` 只保留有价值的测试与探针
- 历史无用产物已清理，不再在根目录长期堆积

如果 `doc/webUI方案集/01-目录结构.md` 与实际结构出现偏差，也需要同步修正。

## 风险

本次重组的主要风险如下：

- 旧本地版迁移后出现相对路径失效
- `.spec` 打包文件路径失效
- README 未同步导致新入口再次混乱
- 删除规则过猛，误删仍有参考价值的文件

为降低风险，本次实施顺序必须是：

1. 先做迁移清单
2. 再检查旧本地版运行/打包路径
3. 再迁移旧本地版
4. 再删除无用文件
5. 最后更新 README 和目录文档

## 验收标准

本次实现完成后，必须满足以下标准：

1. Web 主线仍保持在 `MetaQA_CloudTraining/`
2. 旧本地版迁移到 `legacy/local_desktop/`
3. 旧本地版迁移后可运行
4. 旧本地版迁移后可按原规则继续打包
5. 根目录明显减少旧入口与杂项文件
6. `test/` 中无明显无用的大产物、缓存和一次性结果文件
7. README 与必要目录文档完成同步

## 下一步

在本设计确认后，下一步只生成实施计划，并按“先迁移主线、再清理杂项、最后
修文档”的顺序执行，不把这轮需求扩大为 Web 版内部结构重构。
