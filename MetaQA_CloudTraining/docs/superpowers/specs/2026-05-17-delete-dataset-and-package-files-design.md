# 删除数据集与产物文件一致性设计

本文档定义当前版本中“删除数据集”和“删除产物”两条链路的文件清理边界，
目标是让删除操作与用户预期保持一致，同时避免误删训练目录和其他无关产物。

## 背景

当前实现中，删除数据集已经会尝试删除对应数据集目录，删除产物也会尝试删除
`packages.file_path` 指向的 zip 文件，但两条链路的删除边界、异常处理和回归
测试还不够明确。用户希望删除动作能够“确实删除其文件”，同时明确选择了保守
边界：删除数据集时不联动删除训练目录和产物包。

## 目标

本设计要达成以下目标：

- 删除数据集时，必须删除该数据集自身目录中的文件。
- 删除产物时，必须删除该产物记录对应的 zip 文件。
- 删除数据集时，不删除训练运行目录，不删除已有产物包。
- 删除产物时，不删除训练目录中的 `best.pt`、`best.onnx`、`results.csv`
  等训练文件。
- 删除逻辑失败时，返回明确结果，不伪装成成功。

## 非目标

本设计不包含以下行为：

- 不重构训练任务删除逻辑。
- 不新增“删除数据集时一并删除训练目录”的联动行为。
- 不新增“删除数据集时一并删除所有相关产物”的联动行为。
- 不处理历史上已经丢失但未同步清理的脏数据迁移。

## 当前实现审计

当前代码中的现状如下：

- `dataset_manager.delete_dataset()` 会：
  - 把 `datasets.status` 设为 `deleted`
  - 删除 `images` 表中的关联记录
  - 删除 `datasets/<ds_id>` 目录
- `package_manager.delete_package()` 会：
  - 删除 `packages.file_path` 指向的 zip 文件
  - 删除 `packages` 表记录

当前问题不在“完全没删文件”，而在于缺少明确契约：

- 哪些路径必须删，哪些不能删，没有统一定义。
- 目标文件不存在时是否算失败，没有清晰规则。
- 缺少针对删除边界的回归测试。

## 设计原则

本次设计遵循以下原则：

1. 删除边界必须保守且明确。
2. 文件删除与记录删除的顺序必须可理解。
3. 目标文件缺失和目标文件删除失败必须区分处理。
4. 不把“顺手清理更多内容”混入本次行为修改。

## 行为设计

本节定义删除数据集和删除产物的最终契约。

### 删除数据集

删除数据集时，系统只处理该数据集本身的目录和数据库记录。

必须执行的动作：

1. 确认目标数据集存在且处于可删除状态。
2. 删除 `images` 表中的关联图片记录。
3. 更新 `datasets` 表状态为 `deleted`。
4. 删除 `datasets/<ds_id>` 目录及其下的图片、标签、`dataset.yaml` 等文件。

明确禁止的动作：

- 不删除该数据集对应训练任务的 `run_dir`
- 不删除 `packages` 表中的产物记录
- 不删除 `packages.file_path` 指向的 zip 文件

### 删除产物

删除产物时，系统只处理该产物记录本身和它指向的 zip 文件。

必须执行的动作：

1. 读取 `packages.file_path`
2. 删除对应 zip 文件
3. 删除 `packages` 表中的产物记录

明确禁止的动作：

- 不删除训练运行目录
- 不删除 `best.pt`
- 不删除 `best.onnx`
- 不删除 `results.csv`
- 不删除与该产物无关的其他 zip 文件

## 异常处理策略

本设计采用“保守一致化”策略，而不是“严格失败一刀切”。

### 目标文件不存在

如果目标目录或目标 zip 文件本来就不存在，系统仍然删除对应数据库记录，
同时记录 warning 日志。这样可以清理历史脏记录，不会让用户卡在“记录删不掉”
的状态。

### 目标文件存在但删除失败

如果目标路径存在，但在删除过程中出现权限错误、文件占用或其他 IO 异常，
系统必须返回失败，不得继续假装删除成功。这样可以避免数据库记录已经删掉，
但磁盘文件还残留的情况。

## 模块设计

本次改动保持模块职责不变，不把删除逻辑塞进 API 层。

### `app/core/dataset_manager.py`

该模块继续负责数据集删除主逻辑。

建议补充的职责：

- 在删除数据集目录前后记录清晰日志
- 区分“目录不存在”和“目录删除失败”
- 让删除返回值能够表达失败场景

### `app/core/package_manager.py`

该模块继续负责产物删除主逻辑。

建议补充的职责：

- 在删除 zip 文件前校验路径
- 区分“文件不存在”和“文件删除失败”
- 仅在文件处理结果满足契约时删除数据库记录

### API 层

API 层保持轻量，不新增业务删除细节，只负责：

- 调用 manager
- 把失败结果转换成明确的 HTTP 响应

## 测试设计

本次改动必须补回归测试，覆盖删除边界。

### 数据集删除测试

至少覆盖以下场景：

1. 删除数据集时，`datasets/<ds_id>` 目录被删除。
2. 删除数据集时，数据库中的 `images` 关联记录被删除。
3. 删除数据集时，不删除无关训练目录。
4. 删除数据集时，不删除已有产物 zip。
5. 当数据集目录缺失时，仍能完成数据库记录清理。
6. 当数据集目录存在但删除报错时，删除操作失败。

### 产物删除测试

至少覆盖以下场景：

1. 删除产物时，zip 文件被删除。
2. 删除产物时，`packages` 记录被删除。
3. 删除产物时，不删除训练目录内容。
4. 当 zip 文件缺失时，仍能删除 `packages` 记录。
5. 当 zip 文件存在但删除报错时，删除操作失败。

## 风险与兼容性

本次改动是行为收紧，不是结构重构，风险集中在异常路径。

主要风险如下：

- 如果删除异常未正确捕获，可能造成 API 返回 500。
- 如果删除顺序处理不当，可能出现“记录没了但文件还在”。
- 如果测试覆盖不够，后续容易再次误改删除边界。

兼容性方面，本设计不会改变用户的主路径和目录布局，只会让现有删除契约更
清晰、更稳定。

## 验收标准

本次实现完成后，必须满足以下验收标准：

1. 删除数据集后，其数据集目录确实不存在。
2. 删除数据集不会影响训练目录和产物 zip。
3. 删除产物后，其 zip 文件确实不存在。
4. 删除产物不会影响训练目录内容。
5. 目标文件缺失时，仍能清理数据库记录。
6. 目标文件存在但物理删除失败时，接口返回失败。
7. 相关回归测试全部通过。

数据迁移。
