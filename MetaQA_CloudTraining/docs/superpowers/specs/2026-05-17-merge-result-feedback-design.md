# 合并结果反馈设计

本文档定义“合并新增数据”成功后的反馈方式。目标是让用户在补传完成后立刻
知道本次到底新增了多少、覆盖了多少，而不是只看到一个模糊的“合并成功”。

## 背景

补传/合并数据集现在已经改为新的业务语义：

- 同名图片直接覆盖
- 同名标签直接覆盖
- 图片和标签独立处理

在这种前提下，合并完成后的结果反馈就变得非常重要。用户需要第一时间确认：

- 这次到底新增了多少图片
- 覆盖了多少旧图片
- 新增了多少标签
- 覆盖了多少旧标签

如果前端不展示这些结果，用户就无法快速判断补传是否按预期生效。

## 目标

本设计要达成以下目标：

- 合并成功后立刻弹出一条明确的成功摘要。
- 合并弹窗内部保留“本次合并结果”展示区，便于回看。
- 只对 `action=merge` 生效，不影响新建数据集导入提示。
- 对缺失字段做前端兜底，避免老响应结构导致页面报错。

## 非目标

本设计不包含以下改动：

- 不修改后端合并逻辑。
- 不新增数据库记录或历史结果持久化。
- 不给新建数据集导入弹窗增加同样的结果卡片。
- 不扩展为全局通知中心或历史日志列表。

## 当前实现审计

当前 `merge-dataset-modal` 在上传完成后，只做了这些事情：

1. 调用 `API.uploadComplete(..., "merge")`
2. 成功后直接关闭弹窗
3. 不展示合并结果的结构化信息

这意味着即使后端已经返回了诸如：

- `images_imported`
- `images_overwritten`
- `labels_imported`
- `labels_overwritten`

前端也没有把这些结果呈现出来。

## 设计原则

本次设计遵循以下原则：

1. 反馈必须即时。
2. 反馈必须能回看。
3. 反馈必须只作用于合并链路。
4. 前端必须兼容字段缺失场景。

## 方案对比

本次确认采用双通道反馈，而不是单一通道。

### 方案 A：弹窗 + 结果区（最终采用）

合并成功后：

- 先弹出一条成功摘要
- 再在合并弹窗内部展示“本次合并结果”区块

优点：

- 用户立刻能看到结果
- 关闭提醒后仍能回看
- 不需要额外页面跳转

缺点：

- 比单一提示多一块展示区域

### 方案 B：只弹窗

优点：

- 实现最简单

缺点：

- 关闭后信息就丢失

### 方案 C：只结果区

优点：

- 信息可回看

缺点：

- 用户可能没有立刻注意到结果

## 最终设计

本次采用 **方案 A**。

### 成功弹窗

当合并完成且接口返回成功时，前端生成一条结构化摘要并立即提示用户。

建议文案格式如下：

```text
合并完成：图片新增 X，图片覆盖 Y，标签新增 A，标签覆盖 B
```

如果后端字段缺失，则对应值按 `0` 展示。

### 弹窗内结果区

合并弹窗在成功后不再立刻丢失这次结果，而是保留一个“本次合并结果”区域。

该区域至少展示四项：

- 图片新增
- 图片覆盖
- 标签新增
- 标签覆盖

建议以紧凑卡片或两列统计条目的形式展示，避免占据过高空间。

### 关闭时机

当前弹窗成功后会立即关闭。为了让“结果区”有意义，需要调整为：

- 合并成功后保留弹窗
- 显示结果区
- 由用户自行关闭

如果后续仍希望自动关闭，可单独讨论，但本设计默认以“结果可回看”为优先。

## 数据设计

前端从 merge 响应中读取以下字段：

| 字段 | 用途 | 缺省值 |
|------|------|--------|
| `images_imported` | 图片新增数 | `0` |
| `images_overwritten` | 图片覆盖数 | `0` |
| `labels_imported` | 标签新增数 | `0` |
| `labels_overwritten` | 标签覆盖数 | `0` |

前端本地维护一个 `mergeResult` 状态对象，例如：

```js
{
  imagesImported: 0,
  imagesOverwritten: 0,
  labelsImported: 0,
  labelsOverwritten: 0,
  summaryText: ""
}
```

## 交互设计

### 合并开始前

- 结果区隐藏
- `mergeResult` 清空

### 合并进行中

- 继续显示上传进度和速度
- 不显示结果区

### 合并成功后

- 上传状态结束
- 解析接口返回结果
- 生成 `summaryText`
- 弹出成功提示
- 在弹窗中展示结果区
- 保持弹窗可关闭

### 合并失败后

- 不显示结果区
- 保留错误提示逻辑

## 模块设计

### `static/js/modals.js`

本次改动主要集中在 `merge-dataset-modal` 组件。

需要增加的职责：

- 保存一次合并结果
- 生成结果摘要文案
- 控制结果区显示/隐藏
- 调整成功后的关闭时机

### 后端

后端不需要为本次需求新增逻辑，只需要继续按既有结构返回 merge 结果字段。

## 测试设计

本次改动必须增加前端契约测试，至少覆盖以下场景：

1. 合并成功后生成正确摘要文案。
2. 缺少任一字段时，摘要按 `0` 兜底。
3. 合并成功后显示结果区。
4. 合并成功后不自动清空结果区。
5. 新建数据集导入成功时不走 merge 结果提示逻辑。

如果当前前端测试基础不足，也可以先从纯函数或小型辅助方法开始，把摘要生成
规则锁住。

## 风险与兼容性

主要风险如下：

- 若仍然在成功后立即关闭弹窗，结果区将没有意义。
- 若前端直接信任字段存在，旧响应结构可能导致 `undefined` 显示。
- 若把逻辑误加到新建导入链路，会改变用户原有体验。

兼容性方面，本设计保持：

- 后端接口不变
- 新建导入链路不变
- 只增强 merge 成功后的反馈方式

## 验收标准

本次实现完成后，必须满足以下标准：

1. 合并成功后会弹出明确摘要。
2. 合并弹窗内部会显示“本次合并结果”区域。
3. 结果区包含四项统计：图片新增、图片覆盖、标签新增、标签覆盖。
4. 字段缺失时按 `0` 兜底，不出现 `undefined`。
5. 新建数据集导入成功时不显示 merge 结果反馈。
6. 相关前端测试通过。

## 下一步

需求扩大到全局消息中心或历史合并记录页。
