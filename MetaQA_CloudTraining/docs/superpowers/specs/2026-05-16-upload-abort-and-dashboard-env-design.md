# 上传取消收尾与概览环境信息修复设计

本次设计覆盖两个相邻问题：一是上传弹窗点击“取消上传”后仍会等待当
前分片传完，二是概览界面的环境信息字段显示为 `-`。目标是在不扩大后
端协议面的前提下，先完成最小可用修复，再定位并修复概览页数据缺口。

## 范围

本次变更只处理以下内容：

- 前端分片上传的当前请求立即中断
- 上传弹窗组件卸载时的请求引用清理
- 概览页环境信息显示为 `-` 的根因定位
- 根因明确后的最小修复与验证

本次变更不处理以下内容：

- 新增上传取消后端接口
- 上传会话后台清理机制重构
- 与概览页无关的系统页字段改版

## 方案选择

推荐采用前端立即中断方案，而不是新增后端取消接口。

- 方案一：前端 `xhr.abort()` 立即中断当前分片
  - 优点：改动小，风险低，交互立即改善
  - 缺点：服务端已创建的上传会话仍保留到后续清理
- 方案二：前后端一起实现取消接口
  - 优点：服务端临时分片更干净
  - 缺点：新增 API、清理逻辑和更多验证面

本次采用方案一。

## 设计细节

上传链路保持现有 `init -> chunk -> complete` 结构不变。`API.uploadChunk()`
返回 Promise 的同时暴露 `abort()` 方法，弹窗组件保存“当前请求”引用。
用户点击“取消上传”时，组件先更新 UI 状态，再调用当前请求的
`abort()`，从而立即中止当前分片上传。组件在请求完成、失败、取消和卸
载时，都统一清理请求引用，避免已销毁组件继续回写状态。

概览页环境信息问题按三层排查：先看 `dashboard` 前端展示字段，再看
`/api/v1/system/status` 返回结构，最后看 `system_manager.py` 的组装逻
辑。若字段名不匹配、值缺失或轻量化改造后未再回填环境摘要，则在最小
责任层修复，优先保持 `main.py` 和主前端编排不增加业务逻辑。

## 验证

验证分为两部分：

- 前端修改后执行编辑器诊断与 `node --check`
- 概览页问题修复后检查接口返回结构与前端绑定字段是否一致
