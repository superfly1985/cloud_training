# 训练实时刷新与自动打包设计

## 背景

当前训练页、训练日志弹窗、损失曲线弹窗和顶部 GPU 状态都缺少足够
频繁的自动刷新。训练完成后，任务状态也可能没有及时从 `running`
或 `pending` 收敛到 `completed`，导致产物包未自动生成，产物管理中
也看不到对应下载项。

本设计采用“前端轮询 + 后端状态收敛 + 自动打包”的方案，目标是在
不引入 WebSocket 的前提下，补齐实时体验和训练完成后的自动产物链路。

## 目标

- 训练任务列表中的进度、状态、指标可实时刷新。
- 训练日志弹窗与损失曲线弹窗在打开期间持续刷新。
- 顶部 GPU 状态栏以高频率刷新。
- 训练完成后自动收敛状态，并自动生成产物包。
- 自动打包内容遵循 `doc/webUI方案集/04-产物包管理.md`。
- 若 ONNX 或 TFLite 转换失败，仍生成基础产物包，并在产物管理中明
  确标记为“未转换”或“部分转换”。

## 范围

本次改动聚焦于以下文件与模块：

- `static/js/training-tab.js`
- `static/js/modals.js`
- `static/js/app.js`
- `app/core/monitor_manager.py`
- `app/core/training_manager.py`
- `app/core/package_manager.py`

本次不引入 WebSocket，不重做整体前端架构。

## 方案概览

系统按以下方式工作：

1. 前端定时轮询训练列表、任务指标、日志、曲线和系统状态。
2. 后端在刷新指标时同步判断任务是否仍在运行、是否已经完成、是否失
   败。
3. 任务首次进入 `completed` 时自动触发产物打包。
4. 打包完成后，产物页可直接展示对应包；如果转换不完整，则在列表和
   详情中明确标记转换状态。

## 前端设计

### 训练列表实时刷新

`training-tab.js` 增加页面级定时器。

- 页面加载后立即拉取一次训练任务列表。
- 每 3 秒重新拉取一次任务列表。
- 对于状态为 `pending` 或 `running` 的任务，追加调用
  `/api/v1/training/{task_id}/refresh`。
- 将刷新结果直接回写到当前列表项，更新：
  - `current_epoch`
  - `box_loss`
  - `cls_loss`
  - `map50`
  - `map50_95`
  - `status`

关闭页面或组件卸载时，必须清理定时器。

### 日志弹窗实时刷新

`modals.js` 中的训练日志弹窗在打开时：

- 立即请求一次 `/api/v1/training/{task_id}/log`
- 每 2 秒轮询一次
- 仅在用户滚动接近底部时自动滚动到底部，避免阅读历史日志时被强制拉
  回末尾

弹窗关闭或组件卸载时清理定时器。

### 曲线弹窗实时刷新

`modals.js` 中的训练曲线弹窗在打开时：

- 立即请求一次 `/api/v1/training/{task_id}/curve`
- 每 3 秒轮询一次
- 若数据为空则显示空状态而不是纯空白画布

弹窗关闭或组件卸载时清理定时器。

### 顶部 GPU 状态刷新

`app.js` 顶部状态栏轮询周期从 30 秒缩短为 3 秒。

高频刷新仅使用现有 `/api/v1/system/status` 接口，不新增前端状态管理
框架。

## 后端设计

### 指标刷新与状态收敛

`monitor_manager.py` 扩展 `refresh_task_metrics(task_id)`：

- 继续从 `results.csv` 提取最新 epoch、loss、mAP。
- 同时检查：
  - `train.log`
  - `run_dir/train/results.csv`
  - `run_dir/train/weights/best.pt`
  - 训练进程是否仍存在

状态收敛规则如下：

- 若进程仍存在，任务保持 `running`
- 若日志包含 `epochs completed` 或权重文件已存在且进程已退出，任务转
  `completed`
- 若进程已退出且日志存在异常，或没有有效产物文件，任务转 `failed`

这样即使 watcher 漏掉状态更新，前端轮询也能把任务拉回正确状态。

### 训练完成后的自动打包

`training_manager.py` 中的训练 watcher 在任务完成时：

- 把状态写为 `completed`
- 调用自动打包入口

`monitor_manager.py` 中的状态收敛也要具备同样能力：

- 当发现任务应转为 `completed` 时，若尚未创建包，则触发自动打包

为避免重复执行，需要按 `task_id` 做幂等校验：如果数据库中已经存在该
任务对应的产物包记录，则直接跳过。

## 产物打包设计

`package_manager.py` 改为支持自动打包，并遵循
`doc/webUI方案集/04-产物包管理.md`。

### 标准打包内容

自动打包目标文件：

- `best.pt`
- `best.onnx`
- `best_fp16.tflite`
- `best_fp32.tflite`
- `results.csv`
- `dataset.yaml`
- `info.json`

若某些转换文件不存在，不阻断 ZIP 生成，但必须在元数据中记录缺失原因
与状态。

### 转换状态

为支持“已转换 / 部分转换 / 未转换”展示，`info.json` 和数据库产物记录
需要包含转换状态信息：

- `conversion_status`
  - `complete`
  - `partial`
  - `not_converted`
- `conversion_items`
  - `onnx`
  - `tflite_fp16`
  - `tflite_fp32`
- `conversion_errors`

判定规则：

- 三个目标文件都成功：`complete`
- 至少一个成功、至少一个失败：`partial`
- 三个都失败或均未生成：`not_converted`

### 产物命名

仍沿用文档中的命名规则：

- ZIP 文件名：`{数据集名}_{版本号}.zip`
- 包内模型文件命名可在复制阶段转为文档要求的带前缀命名

## 前端产物展示设计

产物管理页与详情弹窗增加转换状态展示。

- 列表卡片新增转换状态标签：
  - `已转换`
  - `部分转换`
  - `未转换`
- 详情页展示每个目标文件的状态：
  - 已生成
  - 转换失败
  - 未执行

如果自动打包只生成了基础文件，用户也能立即下载该 ZIP。

## 错误处理

系统必须保证以下行为：

- 训练完成但转换失败时，不影响基础包入库
- 自动打包失败时，训练状态仍保持 `completed`，但写入日志并在 UI 中显
  示为“打包失败”
- 指标刷新失败时，不要把正常任务误判成失败
- 轮询接口失败时，前端保留上一次成功数据显示，不闪空

## 验证策略

需要覆盖以下验证点：

1. 训练页列表在训练过程中自动更新 epoch 与状态。
2. 日志弹窗打开后持续追加新日志。
3. 曲线弹窗打开后可看到持续增长的曲线数据。
4. GPU 状态栏刷新周期明显缩短。
5. 训练完成后任务自动转为 `completed`。
6. 训练完成后自动生成产物包。
7. 转换失败时仍能看到 ZIP，且在产物页中明确显示“未转换”或“部分转
   换”。

## 实施顺序

建议按以下顺序落地：

1. 后端状态收敛与自动打包
2. 前端训练页实时刷新
3. 日志与曲线弹窗实时刷新
4. 顶部 GPU 状态高频刷新
5. 产物页转换状态展示

## 风险

本方案的主要风险如下：

- 高频轮询可能增加接口请求量，但在当前项目规模下可接受
- 自动打包链路增加了训练完成后的收尾复杂度，需要做好幂等控制
- 转换逻辑依赖外部环境，必须允许部分成功而不阻断基础包生成

## 结论

本方案在不引入 WebSocket 的前提下，能补齐训练实时监控、状态收敛和
自动产物生成三个关键缺口。它与当前项目结构兼容，实施成本可控，适合
当前阶段直接落地。
