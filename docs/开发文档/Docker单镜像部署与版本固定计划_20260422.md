# Docker 单镜像部署与版本固定计划（Ubuntu 22.04）

## 一、结论摘要
- 当前问题的根因是“运行时环境漂移”（典型如 `protobuf` 被导出链路反向升级），不是单点代码缺陷。
- 采用 `Ubuntu 22.04 + Docker 单镜像 + 版本锁定`，可以把训练、ONNX 导出、TFLite(FP32/FP16) 统一到可复现环境中，显著降低维护成本。
- 首版建议先做“单镜像稳定版”，在镜像内固定关键依赖版本，并在导出阶段强制 CPU，优先保证结果稳定与可回滚。

## 二、目标与范围
- 目标：
  - 一套镜像同时完成：训练、ONNX 导出、TFLite FP32 导出、TFLite FP16 导出。
  - 彻底禁止运行时自动拉包，杜绝训练中途依赖漂移。
  - 新机器拉取同一镜像后结果一致，支持版本化回滚。
- 范围：
  - 服务器基础系统：Ubuntu 22.04（干净系统）。
  - 部署方式：Docker 镜像（单镜像方案）。
  - GUI 侧保持现有流程，后续改为“调用容器执行训练脚本”。

## 三、设计原则
- 固定版本优先：关键库全部显式固定，不使用浮动版本范围。
- 构建期安装、运行期只执行：所有依赖在镜像构建时完成安装，运行期不执行 `pip install`。
- 训练与导出同镜像、导出强制 CPU：降低 TFLite 导出的偶发波动。
- 可观测性：保留关键版本日志与导出结果日志，便于问题复盘。

## 四、推荐版本矩阵（v1.1 推荐冻结版）
> 说明：基于“干净 Ubuntu 22.04”前提做了二次优化，原则是“最小依赖可复现”。先冻结核心必装层，再按需叠加可选层。

### 4.2 可选扩展层（非核心，按需安装）

| 组件 | 建议版本 | 何时启用 |
|---|---|---|
| onnxruntime-gpu | 1.22.0 | 容器内需要额外 ONNX 推理验证时 |
| tflite_support | 不默认安装 | 仅在明确依赖其 API 时启用 |

### 4.3 明确优化点
- 默认不安装 `onnxruntime-gpu`、`tflite_support`，先收敛冲突面再逐步加功能。
- 严禁在运行期安装或升级依赖，所有版本变更只能通过镜像重建发布。
- `protobuf` 固定为 `5.29.5`，避免再次出现训练后版本被动漂移。

## 五、单镜像实施方案
1. 基础镜像选择
   - 以带 CUDA 运行时的 Ubuntu 22.04 镜像为基础。
   - 固定 `TZ=Asia/Shanghai`，统一日志与命名时间戳语义。
2. 镜像构建
   - 使用 `requirements.lock.txt`（全量显式版本）进行安装。
   - 构建末尾执行 `pip check` 和版本打印，写入镜像构建日志。
3. 运行环境约束
   - 设置：
     - `YOLO_AUTOINSTALL=0`
     - `ULTRALYTICS_AUTOINSTALL=0`
     - `PYTHONDONTWRITEBYTECODE=1`
   - 禁止训练脚本里任何动态安装动作。
4. 导出策略
   - ONNX 导出：作为必需产物。
   - TFLite 导出：在同镜像内执行，但导出阶段强制 CPU（`CUDA_VISIBLE_DEVICES=-1`）。
   - 目标产物：`*_float32.tflite`、`*_float16.tflite` 同时存在。
5. 产物与命名
   - 延续当前规则：`命名_时间戳`（东八区）。
   - ZIP 内保留：`pt/onnx/tflite_fp32/tflite_fp16/args/dataset`。

## 六、验证与验收标准
- 环境验收（构建后）
  - 关键包版本逐项比对，必须与锁定版本一致。
  - `pip check` 无冲突。
  - `pip freeze` 输出写入制品，作为发布基线存档。
- 功能验收（每次镜像发布）
  - 同一数据集连续训练 3 轮：
    - 训练成功。
    - ONNX 导出成功。
    - TFLite FP32/FP16 均成功。
    - 产物 ZIP 完整，命名符合规则。
- 稳定性验收
  - 在新开一台 Ubuntu 22.04 服务器，仅通过拉取镜像即可复现同样流程。

## 七、发布与回滚策略
- 镜像 Tag 规范（建议）
  - `vX.N.M`（如：`v2.3.0`）
  - 预发布：`vX.N.M-rc1`
- 发布流程
  - 构建镜像 -> 验证脚本 -> 回归测试 -> 打 Tag -> 推送镜像仓库。
- 回滚流程
  - 发现异常时直接切回上一个稳定 Tag，无需服务器重新修环境。

## 八、风险与对策
- 风险：上游依赖行为变化（即使同版本也可能重新发布轮子）。
  - 对策：固定镜像 Digest + 私有镜像仓库缓存。
- 风险：GPU/驱动与容器 CUDA 兼容性差异。
  - 对策：先做一轮“宿主机驱动-容器 CUDA”兼容性自检。
- 风险：单镜像体积较大。
  - 对策：后续可拆分为“训练镜像/导出镜像”双镜像方案（当前阶段先保证稳定）。

## 九、执行里程碑（建议）
- M1：产出并评审 `Dockerfile + requirements.lock.txt + verify_env.py`。
- M2：完成首个镜像构建，跑通 1 轮端到端训练导出。
- M3：完成 3 轮稳定性回归，冻结首个稳定 Tag。
- M4：GUI 切换为调用容器流程，停用服务器动态修复逻辑。
