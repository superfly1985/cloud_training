# 训练后自动导出 ONNX/TFLite 实施计划

## 1. 目标
- 在现有训练流程不被打断的前提下，训练完成后自动导出 `.onnx` 与 `.tflite`。
- 保持 `.pt` 仍为主产物，导出失败不影响训练成功判定。
- 日志可清晰看到导出结果、文件路径、失败原因。

## 2. 范围
- 修改文件：`cloud_training_gui.py`
- 不改动核心训练参数逻辑，只追加“训练完成后导出”步骤。
- GUI 层先不新增复杂设置项，先采用“ONNX默认导出 + TFLite可选尝试”的保守策略。

## 3. 当前已知环境事实
- 当前服务器可稳定训练（`/root/miniforge3/bin/python3` + `torch 2.6.0+cu124`）。
- 当前模型加载兼容问题已通过环境变量方案规避。
- 数据集路径问题已有训练前自动校正逻辑。

## 4. 实施方案（明天执行）
1. 在训练脚本模板中加入导出函数：
   - 成功训练后定位 `best.pt`（优先 `weights/best.pt`，其次 `last.pt`）。
   - 执行 `model.export(format="onnx")`。
   - 执行 `model.export(format="tflite")`（异常捕获，不中断主流程）。
2. 日志规范：
   - 输出导出开始、导出成功、输出文件路径。
   - 输出失败时的异常摘要（控制长度，避免刷屏）。
3. GUI 日志对齐：
   - 在“最终训练日志”后补一段“导出结果摘要”。
4. 失败降级策略：
   - ONNX 失败：标记警告，训练仍视为成功。
   - TFLite 失败：给出“依赖缺失/版本冲突”提示，训练仍视为成功。

## 5. 验收标准
- 训练成功后目录下至少保留：
  - `best.pt`
  - `best.onnx`（若导出成功）
  - `best*.tflite`（若导出成功）
- GUI 日志可看到明确结果：
  - `ONNX 导出成功/失败`
  - `TFLite 导出成功/失败`
- 任何导出失败都不会把训练状态改成失败。

## 6. 风险与处理
- 风险1：TFLite 依赖不足（常见）。
  - 处理：将 TFLite 作为可选导出，失败只告警。
- 风险2：不同 Ultralytics 版本导出参数差异。
  - 处理：优先使用最简导出调用，避免激进参数。
- 风险3：导出耗时较长。
  - 处理：日志提示“导出中”，避免误判卡死。
